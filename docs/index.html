Lock distribuído para Laravel, com uma chave e uma seção crítica

PzMonitor é um wrapper fino e estático sobre Cache::lock: garante que apenas um processo execute um trecho por vez, libera o lock sempre e mantém as mensagens de erro que a aplicação já conhece.

Cache::lock Redis 100% estático Owner-safe Zero primitivo reimplementado

O que é o PzMonitor?

O PzMonitor implementa exclusão mútua distribuída sobre o lock de cache do Laravel. A API é a de um monitor por chave única: você nomeia um processo por uma chave única e o pacote garante que apenas um executor entre por vez, entre processos, filas e máquinas.

Nada reimplementado

Aquisição atômica, owner token e liberação condicionada ao dono são todos do framework (RedisLock).

Liberação garantida

Os métodos com callback liberam o lock em finally — mesmo quando o callback lança exceção.

Perdedor não atrapalha

Quem falha na aquisição (busy ou timeout) nunca libera nem invalida o lock do processo detentor.

Laravel pronto

Auto-discovery do Service Provider, config publicável e prefixo de chave próprio (pzmonitor:).

O ponto de entrada é sempre a classe estática:

use Puzl\PzMonitor\PzMonitor;

PzMonitor::lock('emissao-nfe:1234', fn () => $this->emitirNfe(1234));

O que faz e o que não faz

✓ O PzMonitor faz

  • Seção crítica com espera (lock) ou falha rápida (tryLock)
  • Lote multi-chave all-or-nothing com ordem fixa (tryLockMany)
  • Handle cru do lock para casos que não cabem em callback (acquire, tryAcquire)
  • Liberação administrativa de chave presa (forceRelease)
  • Prefixo de chave configurável, isolando o Monitor da aplicação
  • Store de cache configurável (PZMONITOR_STORE)
  • Exceções tipadas com a chave do processo em propriedade readonly

✗ O PzMonitor não faz

  • Renovar o TTL — não há watchdog; callback mais longo que o TTL perde o lock
  • Reentrância — lock aninhado na mesma chave bloqueia até o timeout
  • Fencing tokens / Redlock multi-nó — é lock de eficiência, não de correção absoluta
  • Condition variables (Wait/Pulse) — esse papel é das queues e events do Laravel
  • Excluir sem cache compartilhado — array e file local não excluem entre máquinas
  • Métricas e eventos — fora do escopo desta versão

Convenção de chave
Use um identificador estável e específico do processo: "emissao-nfe:{$pedidoId}", "sincronizacao-estoque:{$lojaId}". O prefixo pzmonitor: é aplicado automaticamente.

API — lock()

PzMonitor::lock(string $uniqueProcessKey, callable $callback, ?int $waitTimeoutInSeconds = null, ?int $lockTtlInSeconds = null): mixed

Seção crítica com espera. Executa o callback com exclusividade e libera o lock sempre — inclusive quando o callback lança exceção. Retorna o valor do callback.

use Puzl\PzMonitor\Exception\PzMonitorTimeoutException;
use Puzl\PzMonitor\PzMonitor;

try {
    $nota = PzMonitor::lock(
        uniqueProcessKey: "emissao-nfe:{$pedidoId}",
        callback: fn () => $this->emitirNfe($pedidoId),
        waitTimeoutInSeconds: 60,   // omitido = config wait_timeout (padrão 60)
        lockTtlInSeconds: 300,      // omitido = config lock_ttl (padrão 300)
    );
} catch (PzMonitorTimeoutException $e) {
    // Espera esgotada: o callback NÃO executou e o lock do dono ficou intacto.
    report($e);
}
Espera exata
A espera máxima é $waitTimeoutInSeconds exato, com retry interno de 250 ms do framework.

API — tryLock()

PzMonitor::tryLock(string $uniqueProcessKey, callable $callback, ?int $lockTtlInSeconds = null): mixed

Seção crítica com falha rápida: chave ocupada lança PzMonitorBusyException imediatamente e o callback não executa.

use Puzl\PzMonitor\Exception\PzMonitorBusyException;
use Puzl\PzMonitor\PzMonitor;

try {
    PzMonitor::tryLock(
        uniqueProcessKey: "sincronizacao-estoque:{$lojaId}",
        callback: fn () => $this->sincronizarEstoque($lojaId),
        lockTtlInSeconds: 300,      // omitido = config lock_ttl (padrão 300)
    );
} catch (PzMonitorBusyException $e) {
    // Já existe outro processo na mesma chave.
    return response()->json(['message' => $e->getMessage()], 409);
}

API — tryLockMany()

PzMonitor::tryLockMany(array $uniqueProcessKeys, callable $callback, ?int $lockTtlInSeconds = null): mixed

Seção crítica multi-chave, all-or-nothing. Deduplica as chaves e adquire em ordem lexicográfica fixa (anti-deadlock entre lotes concorrentes). Qualquer chave ocupada: libera as já adquiridas (rollback) e lança busy nomeando a chave conflitante.

O problema que ele resolve

Os outros métodos travam uma chave. Mas há operações que tocam duas ou mais entidades ao mesmo tempo, e travar só uma não protege nada.

O caso canônico é a transferência entre contas: transferir(A → B) debita A e credita B. Travando apenas a conta A, nada impede uma transferência X → B simultânea de bagunçar o saldo de B. A operação precisa das duas chaves ao mesmo tempo.

A tentativa ingênua é aninhar — e é exatamente aqui que nasce o deadlock:

// NÃO faça isso.
PzMonitor::tryLock("conta:{$origemId}", function () use ($destinoId) {
    PzMonitor::tryLock("conta:{$destinoId}", fn () => $this->transferir(...));
});

O deadlock que ele evita

Duas transferências simultâneas, em sentidos opostos:

Processo 1: transferir(conta:5 → conta:9)     Processo 2: transferir(conta:9 → conta:5)
  trava conta:5  ✓                              trava conta:9  ✓
  quer conta:9 ... ocupada pelo P2               quer conta:5 ... ocupada pelo P1

Cada um segura o que o outro precisa. Abraço mortal: nenhum dos dois solta, e ambos ficam presos até o TTL expirar.

A solução é a ordem fixa. O tryLockMany ordena as chaves lexicograficamente antes de adquirir. Assim os dois processos, mesmo pedindo em sentidos opostos, pedem na mesma ordem: 

Processo 1: pede [conta:5, conta:9]           Processo 2: pede [conta:5, conta:9]
  trava conta:5  ✓                              tenta conta:5 → OCUPADA → falha limpa
  trava conta:9  ✓
  transfere

Um ganha; o outro leva PzMonitorBusyException na hora, sem ter travado nada. Deadlock impossível por construção — o sort() não é estética, é o mecanismo.

All-or-nothing: o rollback

O callback só executa com o conjunto completo na mão:

pede [conta:5, conta:9, conta:12]
  conta:12  ✓ adquirida     // ordem de string: "conta:12" < "conta:5" < "conta:9"
  conta:5   ✓ adquirida
  conta:9   ✗ ocupada  →  libera conta:12 e conta:5, lança busy nomeando "conta:9"

Sem esse rollback sobrariam duas chaves órfãs presas até o TTL, bloqueando outras operações à toa. O finally do método cobre os dois caminhos: o rollback da aquisição parcial e a liberação normal após o callback (inclusive quando o callback lança exceção).

A ordem não é a que você escreveu
No exemplo acima conta:12 vem antes de conta:5 porque a comparação é de string, não numérica. Isso não afeta a correção — o que importa é que todos os processos usem o mesmo critério — mas explica por que a ordem de aquisição difere da ordem do array.

Deduplicação

["conta:5", "conta:5"] significaria travar a mesma chave duas vezes. Como o lock não é reentrante, a segunda tentativa encontraria a chave ocupada… pelo próprio processo. Auto-deadlock. O array_unique elimina isso — útil quando as chaves vêm de um loop e origem e destino podem coincidir (transferência de uma conta para ela mesma).

Exemplo completo

use InvalidArgumentException;
use Puzl\PzMonitor\Exception\PzMonitorBusyException;
use Puzl\PzMonitor\PzMonitor;

try {
    PzMonitor::tryLockMany(
        uniqueProcessKeys: ["conta:{$origemId}", "conta:{$destinoId}"],
        callback: fn () => $this->transferir($origemId, $destinoId, $valor),
        lockTtlInSeconds: 300,      // omitido = config lock_ttl (padrão 300)
    );
} catch (PzMonitorBusyException $e) {
    // $e->uniqueProcessKey → a chave que estava ocupada.
    report($e);
} catch (InvalidArgumentException $e) {
    // Lista de chaves vazia.
    report($e);
}
TTL do lote
Um único TTL vale para todas as chaves, e o relógio de cada uma começa a correr no momento em que ela é adquirida — não quando o callback começa. Num lote de 50 chaves cujo callback leva 4 minutos, um TTL de 300 s fica apertado: as primeiras chaves já gastaram tempo esperando as outras 49 serem travadas. Dimensione pelo tempo total do lote, não pelo tempo de processar um item — e como o padrão da config raramente basta para lote, este é o caso típico de passar lockTtlInSeconds explícito.

Quando usar cada método

SituaçãoMétodo
Uma rotina, uma chavetryLock() / lock()
Uma operação que toca N entidades e precisa de todastryLockMany()
Precisa esperar em vez de falharlock() — não há versão bloqueante multi-chave
Por que não existe um lockMany() bloqueante
É deliberado. Esperar por múltiplas chaves reintroduz a chance de deadlock que a falha rápida elimina: dois lotes com interseção parcial podem esperar um pelo outro indefinidamente. Se a operação é importante, o padrão é falhar rápido e tentar de novo depois (retry na fila), não esperar segurando locks.

API — acquire()

PzMonitor::acquire(string $uniqueProcessKey, ?int $waitTimeoutInSeconds = null, ?int $lockTtlInSeconds = null): Illuminate\Contracts\Cache\Lock

Aquisição crua bloqueante. Devolve o handle do lock do framework; o caller deve liberar em finally.

use Puzl\PzMonitor\Exception\PzMonitorTimeoutException;
use Puzl\PzMonitor\PzMonitor;

try {
    $lock = PzMonitor::acquire(
        uniqueProcessKey: "fechamento-caixa:{$caixaId}",
        waitTimeoutInSeconds: 60,   // omitido = config wait_timeout (padrão 60)
        lockTtlInSeconds: 300,      // omitido = config lock_ttl (padrão 300)
    );
} catch (PzMonitorTimeoutException $e) {
    report($e);

    return;
}

try {
    $this->fecharCaixa($caixaId);
} finally {
    $lock->release();
}

API — tryAcquire()

PzMonitor::tryAcquire(string $uniqueProcessKey, ?int $lockTtlInSeconds = null): Illuminate\Contracts\Cache\Lock

Aquisição crua com falha rápida. Chave ocupada lança PzMonitorBusyException; o caller deve liberar em finally.

use Puzl\PzMonitor\Exception\PzMonitorBusyException;
use Puzl\PzMonitor\PzMonitor;

try {
    $lock = PzMonitor::tryAcquire("importacao-xml:{$arquivoId}", lockTtlInSeconds: 300);
} catch (PzMonitorBusyException $e) {
    report($e);

    return;
}

try {
    $this->importar($arquivoId);
} finally {
    $lock->release();
}
Prefira o callback
lock(), tryLock() e tryLockMany() já liberam o lock em finally internamente. Use o handle cru só quando a seção crítica não couber em um callback.

API — forceRelease()

PzMonitor::forceRelease(string $uniqueProcessKey): void

Liberação administrativa: ignora o dono do lock. Existe para destravar chave presa por processo morto antes do TTL expirar.

use Puzl\PzMonitor\PzMonitor;

// Ex.: comando artisan de suporte, executado por operação.
PzMonitor::forceRelease("emissao-nfe:{$pedidoId}");
Nunca em fluxo de negócio
Se o dono legítimo ainda estiver dentro da seção crítica, forceRelease() quebra a exclusão mútua e deixa dois processos executando o mesmo trecho. Uso exclusivo de operação/suporte.

Integração Laravel

O PzMonitorServiceProvider é registrado por auto-discovery (declarado em composer.json, extra.laravel.providers). Ele não registra bindings: PzMonitor é estática e lê a config pela Facade Config. O provider apenas:

  • register() — mescla config/pzmonitor.php em config('pzmonitor.*'), sem sobrescrever o que a aplicação já definiu.
  • boot() — registra o publishes com a tag pzmonitor-config, apenas quando rodando em console.
php artisan vendor:publish --tag=pzmonitor-config

Exemplo dentro de um job de fila:

use Puzl\PzMonitor\Exception\PzMonitorBusyException;
use Puzl\PzMonitor\PzMonitor;

class SincronizarEstoqueJob implements ShouldQueue
{
    public function handle(): void
    {
        try {
            PzMonitor::tryLock("sincronizacao-estoque:{$this->lojaId}", fn () => $this->sincronizar());
        } catch (PzMonitorBusyException $e) {
            // Outra execução já está sincronizando esta loja: descarta silenciosamente.
            $this->delete();
        }
    }
}

Configuração (config/pzmonitor.php)

return [
    'store'        => env('PZMONITOR_STORE'),                // null = store padrão da app
    'prefix'       => env('PZMONITOR_PREFIX', 'pzmonitor:'), // prefixo de toda chave
    'wait_timeout' => (int) env('PZMONITOR_WAIT_TIMEOUT', 60),
    'lock_ttl'     => (int) env('PZMONITOR_LOCK_TTL', 300),
];
VariávelPadrãoDescrição
PZMONITOR_STORE null Store de cache dos locks. null usa o store padrão da aplicação. Em produção deve resolver para um backend compartilhado (Redis).
PZMONITOR_PREFIX pzmonitor: Prefixo aplicado a toda chave de lock, garantindo que as chaves do PzMonitor nunca colidam com outras chaves da aplicação.
PZMONITOR_WAIT_TIMEOUT 60 Segundos que lock() e acquire() esperam pela chave antes de lançar PzMonitorTimeoutException.
PZMONITOR_LOCK_TTL 300 Segundos até o cache expirar a chave sozinho, mesmo sem release().

Precedência dos tempos

Os parâmetros $waitTimeoutInSeconds e $lockTtlInSeconds resolvem em três níveis: argumento da chamada > config/pzmonitor.php > padrão do pacote (60 s / 300 s). Omitir o argumento — ou passar null — cai para a config; um valor explícito vence a config, inclusive numa aplicação que a customizou. 

PzMonitor::lock('fechamento-mensal', $callback);                        // usa a config
PzMonitor::lock('fechamento-mensal', $callback, lockTtlInSeconds: 1800); // vence a config
O lock_ttl da config é um piso, não uma solução. O TTL adequado depende da duração de cada rotina, não da aplicação: fechamento de mês e envio de e-mail não querem o mesmo número. Rotina longa deve passar o TTL explícito na chamada.

Exceções

Todas em Puzl\PzMonitor\Exception\. Telas e logs da aplicação consumidora dependem desses textos e não devem ser traduzidos.

ExceçãoQuandoMensagem
PzMonitorException Base abstrata da família; estende \Exception e expõe a propriedade readonly uniqueProcessKey. —
PzMonitorBusyException tryLock, tryLockMany e tryAcquire com a chave ocupada. Already exists process. Process {chave}
PzMonitorTimeoutException lock e acquire com a espera esgotada. getPrevious() devolve o LockTimeoutException do framework. Maximum wait time reached. Process {chave} 
use Puzl\PzMonitor\Exception\PzMonitorException;

try {
    PzMonitor::tryLock('minha-chave', $callback);
} catch (PzMonitorException $e) {
    // Captura busy e timeout de uma vez.
    logger()->warning($e->getMessage(), ['key' => $e->uniqueProcessKey]);
}

tryLockMany() lança ainda InvalidArgumentException quando a lista de chaves é vazia.

Limites e avisos

TTL é rede de segurança, não watchdog
O TTL (padrão 300s) protege contra processo morto (kill -9, OOM). Se a seção crítica passar do TTL, o lock expira e outro processo entra — dimensione $lockTtlInSeconds pelo pior caso da operação.
Não reentrante
Chamar um método do PzMonitor aninhado na mesma chave bloqueia até o timeout (lock/acquire) ou falha imediatamente (tryLock/tryAcquire). Reentrância está fora do escopo.
Exclusão real exige cache compartilhado
A exclusão só vale entre processos que usam o mesmo store. file exclui apenas dentro de uma máquina; array apenas dentro do processo. Em produção, aponte para Redis.
Lock de eficiência, não de correção absoluta
Sob failover de Redis single-instance, dupla aquisição é raramente possível (Kleppmann). Aceitável para os usos atuais; fluxos correção-críticos exigem outra abordagem.
Não configure lock_connection
Enquanto existir call site na aplicação, config/cache.php não deve definir lock_connection — o v1 depende da colisão de chave na conexão cache.

Fora do escopo (upgrade path, sem prazo)

Watchdog de renovação de TTL, reentrância por contador de aquisições, fencing tokens / Redlock multi-nó, condition variables (Wait/Pulse) e métricas/eventos não fazem parte desta versão.

Boas práticas

  • Chave por processo, não por classe. Inclua o identificador do recurso: "emissao-nfe:{$pedidoId}", não "emissao-nfe".
  • Seção crítica curta. Deixe fora do callback tudo que não precisa de exclusividade (validação, montagem de payload).
  • TTL pelo pior caso. Meça a operação mais lenta em produção e some folga.
  • Trate a exceção onde ela significa algo. Busy geralmente é "já está rodando, ignore"; timeout geralmente é fila ou incidente.
  • Handle cru sempre com finally. Sem isso, a chave só libera no TTL.
  • Ordem fixa em multi-chave. Use tryLockMany() em vez de encadear tryAcquire() manualmente — a ordenação lexicográfica é o que evita deadlock entre lotes.

Testes

composer test
composer test:coverage

As suítes Unit e Features rodam no driver de cache array (ArrayStore implementa LockProvider) — sem Redis, Docker ou qualquer infraestrutura externa.

A suíte Integration é a única que verifica a promessa central do pacote — exclusão mútua entre processos — disparando processos PHP concorrentes contra um Redis real. Ela se pula sozinha quando não há Redis alcançável, então localmente basta subir um: 

docker run -d --rm -p 6379:6379 redis:7-alpine
composer test

Variáveis: REDIS_HOST, REDIS_PORT, REDIS_PASSWORD. O workflow .github/workflows/tests.yml sobe o serviço em cada push e pull request, e falha se a suíte de integração for pulada — teste que não roda não protege nada.

Perguntas frequentes

Posso usar junto com outro Monitor da aplicação?

Sim. O prefixo próprio (pzmonitor:) garante que as chaves nunca colidem. Migre por família de chave, nunca por arquivo.

Por que a classe é 100% estática?

Não há estado nem variação de motor: existe um único backend possível (Cache::lock). Adapter ou factory seriam abstração especulativa.

O que acontece se o callback lançar exceção?

O lock é liberado de qualquer forma (finally) e a exceção sobe para o caller sem alteração.

Preciso de Redis para os testes?

Para as suítes Unit e Features, não: o driver array implementa LockProvider e cobre toda a semântica da API num único processo. Para a suíte Integration, sim — ela existe justamente porque o store array não prova exclusão entre processos. Sem Redis ela se pula; no CI ela roda e o job falha se for pulada.
